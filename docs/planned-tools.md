# Planned orchestration tools

This document describes the second iteration of the orchestration layer. These
helpers continue the same design principle as the first PR: they compose the
existing Froide REST API and do not require Django/Froide backend changes.

## New tools

### `draft_followup_for_request`

Fetches a request, inspects its latest message and current status, and returns
an operator-facing draft follow-up without sending anything.

**Returns:** `request_id`, `label`, `status`, `subject`, `draft_message`,
`latest_message_excerpt`, `why`

---

### `preflight_request_submission`

Runs MCP-side checks before `make_request`:

- required subject/body
- short-subject warning
- short-body warning
- light heuristic for whether the body contains an explicit request
- public body existence check
- optional law and campaign lookup

**Returns:** `ok`, `issues[]`, `warnings[]`, `request_preview`, `next_step`

---

### `get_request_analytics`

Builds analytics from `triage_my_requests` results:

- total count
- `status_counts`
- `priority_bands` (`critical`, `high`, `medium`, `low`)
- `top_requests`

No server-side aggregation endpoint is required.

---

### `draft_request`

Creates a suggested subject and body from:

- target public body
- the user's goal
- a description of the records/information sought

This is intended to pair with `preflight_request_submission` and `make_request`.

---

### `followup_after_deadline`

Focuses on `awaiting_response` requests and pairs each candidate with a draft
follow-up generated by `draft_followup_for_request`.

Because the current API surface exposed in this repo does not contain an
explicit deadline field, this tool currently models “after deadline” as
"likely follow-up candidate" rather than a strict legal-deadline computation.

## Composition model

These tools are designed to work together:

1. `draft_request`
2. `preflight_request_submission`
3. `make_request`
4. `triage_my_requests` / `find_requests_needing_action`
5. `draft_followup_for_request` / `followup_after_deadline`
6. `send_followup`

## Next likely refinement

If Froide later exposes explicit deadline metadata, `followup_after_deadline`
can be upgraded from heuristic routing to precise deadline-based filtering
without changing the surrounding MCP workflow.
